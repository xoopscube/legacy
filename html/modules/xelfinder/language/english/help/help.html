Overview

X-elFinder file manager allows you to organize files in a folder hierarchy and configure permissions for individual users and groups.

The combination of useful features for effective collaboration, integration of open source code editors and online image editing tools, allows you to streamline the content creation process. In addition, you have several options for scaling storage. So you can easily and quickly increase your organization's data capacity using cloud storage, Drive, Dropbox, etc.

Usage

Installing X-elFinder will automatically replace the default image manager with the module's file management interface.

Notes on installation

Set default permission for new Files and subdirectories. The following directories require write permissions, such as 777 or 707, to create new files :

  • html/modules/xelfinder/cache
  • html/modules/xelfinder/cache/tmb
  • trust_path/uploads/xelfinder

Upload Files

This module uploads files directly from your computer using browse or drag and drop. You can create new folders to organize files that can be shared and accessed from other modules, such as D3Forum and Pico.

Working with files

Files can be searched by keyword as well as filtered by upload location. You can also select multiple files using the checkboxes, and then download or delete.

You can define permissions, mime types and folders for different user groups in the module's preferences.

Collaborate on files

File manager brings together useful features for you and your team to work together faster and easier.

  • Allow multiple formats
  • Auto-save feature
  • Automatic resizing of uploaded images
  • Create folders per user
  • Configure folder access
  • Share files with your community
  • Share files and folders with selected groups

While your server offers important safeguards such as persistent file storage, you can add new remote storage spaces to expand and increase the performance of shared files.

For example, you may want to share data with other users and at the same time secure a central repository for documents and data. To do this, multiple users need access to the same file system. You can create a file system that combines remote storage systems (cloud storage) into a single namespace. This allows you to scale file storage easily, without downtime or overflowing your server's storage capacity.

Scalability

Enable cloud storage to expand your storage capacity. It's as easy as adding your cloud API key to the module preferences.

Effective segmentation is essential for scaling. You can plan to segregate sensitive and proprietary information within your organization, such as competitor product plans or financial data, individual files or directories to prevent unauthorized access.

At the same time, your less sensitive open-access or public collaboration files are stored in the cloud. These files are always up-to-date and accessible on any device to users with permissions. This results in increased performance capabilities and allows your organization to do more in less time while reducing operational costs. Content segmentation and separation are very effective strategies an organization can implement to prevent unintended consequences.

Store the API key and secret of your cloud services directly in your module preferences for easy access.

Using OAuth 2.0 for Web Server Applications

This document explains how web server applications use Google API Client Libraries or Google OAuth 2.0 endpoints to implement OAuth 2.0 authorization to access Google APIs.

OAuth 2.0 allows users to share specific data with an application while keeping their usernames, passwords, and other information private. For example, an application can use OAuth 2.0 to obtain permission from users to store files in their Google Drives.

This OAuth 2.0 flow is specifically for user authorization. It is designed for applications that can store confidential information and maintain state. A properly authorized web server application can access an API while the user interacts with the application or after the user has left the application.

Web server applications frequently also use service accounts ‚≠ß to authorize API requests, particularly when calling Cloud APIs to access project-based data rather than user-specific data. Web server applications can use service accounts in conjunction with user authorization.

How do I authorize Google API?

All requests to the API must be authorized by an authenticated user.

Both Sign In With Google and One Tap authentication include a consent screen which tells users the application requesting access to their data, what kind of data they are asked for and the terms that apply.

If you add sensitive or restricted ( üîí ) scopes, like scopes that let you access a user's emails or contacts, you will need to submit your app for verification. FAQ to learn more about verifying OAuth scopes ‚≠ß

At a high level, you follow five steps:

  1. Obtain OAuth 2. 0 credentials from the Google API Console.
  2. Obtain an access token from the Google Authorization Server.
  3. Examine scopes of access granted by the user.
  4. Send the access token to an API.
  5. Refresh the access token, if necessary.

Root folder ID

You can set the root_folder_id for X-elFinder. This is the directory (identified by its Folder ID) considered to be the root of your drive.

Normally you will leave this blank and X-elFinder will determine the correct root to use itself.

However, you can set this to restrict X-elFinder to a specific folder hierarchy or to access data within the "Computers" tab on the drive web interface (where files from Google's Backup and Sync desktop program go).

In order to do this you will have to find the Folder ID of the directory you wish X-elFinder to display.

What is the Folder ID Google Drive?

This will be the last segment of the URL when you open the relevant folder in the drive web interface.

So if the folder you want X-elFinder to use has a URL which looks like in the browser:

https://drive.google.com/drive/folders/1XyfxxxxxxxxxxxxxxxxxxxxxxxxxXCL

then you use the "id" as the "root folder_id" in the settings of Google Drive API.

1XyfxxxxxxxxxxxxxxxxxxxxxxxxxXCL

NB. folders under the "Computers" tab seem to be read only (drive gives a 500 error)

There doesn't appear to be an API to discover the folder IDs of the "Computers" tab - please contact us if you know otherwise!

Note also that X-elFinder can't access any data under the "Backups" tab on the Google Drive web interface.

My Drive

Each user has a "root" folder called "My Drive" that functions as their primary hierarchy, and consists of everything that descends from this root folder. The user is the primary owner of this folder.

Shared drives

A shared drive is an organizational structure within Google Drive that lives parallel to My Drive. You can organize an individual file within a shared drive or in My Drive but not both.
