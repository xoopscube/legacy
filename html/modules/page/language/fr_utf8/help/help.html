Overview

The Pico module is one of the most useful and versatile modules. You can use it for content creation, to manage dynamic and static content.

Examples of content creation include documents, articles, pages with custom components and forms, images and videos. Static content is any content, or unchangeable files, that can be delivered to users without processing. For example, web documents or html files, brochures, reports or marketing materials.

Pico is a Trust module based on duplicatable D3 specs which means you can install and run multiple highly customized instances from a single source. Instances such as Content Management System (CMS), Knowledge Management or Wrapper for static content.

The most significant benefits of using Pico coupled with CKEditor and X-elFinder are simplified installation and updates, granular permissions and digital content management.

Usage

D3 modules are designed to let you create categories and contents on the front end. These modules also provide granular user permissions, allowing you to customize access levels to contents and features. You can identify and create user roles which define what a user can see and do.

Contents Management

Back-end

The main administrative view provides the following functionality to let you manage contents and permissions :

  • Batch actions
  • Edit title
  • Virtual path
  • Weight (order)
  • Visible (published)
  • Show in Navigation
  • Show in Menu
  • Allow comments
  • User groups and granular permissions
Front-end

You can create and edit categories and contents directly from the front-end :

  • Visible (published)
  • Show in navigation
  • Show in menu
  • Allow comments
  • Use cache (enable only for static content)
  • Lock (only moderators can edit/delete)
  • Set filters
  • Access revision history
Categories and contents can be moved or exported to other active instances. You can also embed content from one instance into another instance and easily include HTML files. See section "Wraps mode" for more details.

Content Workflow

The module provides a simple publication workflow implementation for creating web content in your organization. This allows you to push a content through predefined states to share information or media with specific audiences. Assuming you have enabled notifications on module preferences, the system notifies the author and the webmasters group, by default.

Content status
Draft or Invisible Content is unpublished or scheduled to be published
Staged or waiting releaseContent is scheduled to be published
PublishedContent is published and available to users
LockedContent is published but only moderators can edit and delete
Archived or ExpiredPreviously published versions of content that has been unpublished and is no longer available

Content Revision history

This feature allows you to view the evolution of the published content. When revision history is enabled, every published change will result in a new version being created. You can see what was revised, who made the revision, compare two revisions, and download changes.

  • View the chronological revision history of the content.
  • See who made changes to the content.
  • Download a revision of the content to reverse any changes.
  • Compare revisions to each other and easily view changes between.

Enable Revision history

To enable this feature, browse to module's preferences :

Content Revision History - Maximum number of revisions per content stored in the database. Default value: 4.
Content Revision History - Minimum time interval for each revision.Default value in seconds: 180.

View Revision History

Revision history gives you a point-in-time snapshot of your content, so you can compare versions and see the date and time they were published. One draft or published version exists at a time, but you can have many archived versions, depending on your preference settings for the maximum number of revisions.

Back-end

In the administrative view "Contents", select "All Contents" to see the revision and status of each content, the date and time of its last modification.

To view an iteration of the content, click on "Edit" and then on "Revision History".

Depending on the status of the content, you can review, copy (save as), modify, publish or unpublish the content.

Front-end

You can compare file revisions using the diff tool associated with Revision History. Assuming your template contains the default variables and a revision exists, it displays the "modified" date and time next to the title :

  • Created - the date and time the content was initially created
  • Modified - the date and time the content was last changed and saved

Regarding the modified content in question, click "Edit content" to view "Revision History" links e.g. refer, download, diff.

Customize Revision History

The revision history uses the Text diff renderer. This is a powerful data comparison tool for parsing strings and reporting differences (additions and deletions) between two revisions. The diff is wrapped in an HTML pre tag, preformatted text is usually rendered using a monospaced font. To highlight the differences and give you control over the layout use the following HTML tags:

del
del is used to represent deleted text.
ins
ins is used to represent the newly added text.

Added and deleted elements are styled using CSS "main.css" and displayed in the template.

You can refine the styles by extending the predefined classes or by modifying an instance of them. 


.pico_history_diff {
    overflow-x: auto;
    white-space: pre-wrap;
    word-wrap: break-word;
    border: var( --border, 1px solid #060E18B3); /* if --border is not defined, fallback */
    padding: 1em;
}
.pico_history_diff del {
    color: var(--del-color, #c62828);
}
.pico_history_diff ins {
    color: var(--ins-color, #388e3c);
    text-decoration: none;
}
You can edit the CSS directly by selecting the module's Templates and the "main.css" file.

Permissions

Only administrators can create subcategories. However, you can assign users and roles to create and mange subcategories (either individually, or for all subcategories on your module instance). For example, you may grant users a specific role with the ability to create content.

Back-end

In the administrative view "Permissions", select a category. The module prompts you to verify the inherited permissions. If you want to customize unique permissions for the category, you can select the checkbox and submit. click on edit.On the back end, select the category and click "Submit" to manage users permissions.

Front-end

On the front end, click "Edit category", then click "Category permissions"

Subcategories and contents inherit permissions from the nearest parent category, in the absence of individual permissions, all inherit from the Top-level category, by default.

Import and Synchronize

The Import tool is intended primarily for bulk data population. This feature resets the instance importing the data, meaning the loss of all existing data and custom settings.

The data synchronization is intended to synchronize incremental changes on the instance deployed.

You can use these tools to populate an instance with data imported from other module instances, and maintain data accuracy. In addition, it is recommended to check that user permissions are set up correctly. This prevents unintended access and ensures that sensitive information is only seen by users with the necessary permissions.

Import data

  1. Select an instance of Pico or TinyD modules.
  2. Click the action button "Import Data".
Warning: All data and settings of the module that imports data will be lost when the process starts.

Synchronize redundant data

Data redundancy occurs when the same piece of data exists in multiple places, whereas data inconsistency is when the same data exists in different formats in multiple tables. Unfortunately, data redundancy can cause data inconsistency, which can result in contents with unreliable information. Data synchronization ensures that all records are consistent.

Run data synchronization to resolve categories or contents issues related to data import.

Clear compiled cache

If you experience rendering problems, for example after a site move or data import, you should clear the cache of compiled templates.

The next time the rendering process is run, the cache for each compiled template is recreated and displayed.

Refresh the web browsers' cache

After clearing the compiled cache, it is strongly recommended clearing web browsers cache !

Custom HTML Head Tags

This module allows you to create custom HTML code for the head and body of module instances, categories and contents.

You can create a custom HTML head for different purposes :

  • Customize a module instance, category or content.
  • Custom CSS stylesheet per module, category or content.
  • Custom JavaScript per module, category or content.
  • Search engine optimization (SEO)
  • Multi-language support

Enable Custom HTML Head

Browse to module's preferences to enable custom HTML header. The following options are available :
Common HTML header [ CSS, JS ]Enter the code or link for the instance of this module.
Enable custom HTML header for each contentYes or No to display front-end textarea to customize each content.
CSS file for this moduleRelative or absolute path can be defined. Default value : {mod_url}/index.php?page=main_css

Custom HTML Head by Category

The front-end provides several options for customising a category. Among these options, the HTML header can override CSS styles.

Category creation and edition
  1. Click "New subcategory" or "Edit category"
  2. Click "Options" (on/off) to override preferences by writing a new line for each option.

For example, override the style to change the background color of content : 

htmlheader:<style>article#pico_container {background:#202f3c;}</style>
Note that this example applies to Themes built with PicoCSS. Other Themes and CSS frameworks may use a different ID.

Custom HTML Head for Content

  1. Content creation and edition
  2. Click HTML headers to display the textarea

For example, this style overrides the category style to change the background color of content :

<style>article#pico_container {background:#202f44;}</style>

Multiple Instances

D3 modules can run multiple instances simultaneously on your web application. Consider the following example :

  • If you have a single instance of Pico, everyone works together in the same place.
  • If you have multiple instances of Pico, users may have access to one or many instances of Pico.
  • You might have instances for different purposes or departments.
  • Each instance is independent with its own blocks, templates, permissions and language
  • Instances can share data, or include content from other instances.

Scalability

A scalable solution must enable organisations to grow appropriately and cost-effectively to meet changing demands.

You can install and customize multiple instances on your web server document root, the public directory of the web server running your web app. Each instance has its own database tables and directory /public_html/modules/instance-name/

Although public instances are independent, they all operate from a single trusted source.
